## 1. Контекст и текущее состояние

### 1.1. Продукт
Telegram-бот [@Photo_2_StickerBot](https://t.me/Photo_2_StickerBot) — превращает фото в стикеры с помощью ИИ (Gemini). Поддерживает:
- 10 групп стилей, ~45 подстилей (аниме, мемы, милые, романтика, мультфильм, игровой, рисунок, манхва, сериалы, русский стиль)
- Эмоции (смех, грусть, злость и т.д.)
- Анимацию/движение (машет рукой, танцует и т.д.)
- Добавление текста на стикер
- Удаление фона (rembg + Pixian)

### 1.2. Текущий сайт
- **Домен:** https://photo2sticker.ru/
- **Стек:** Vite + React + Tailwind CSS + shadcn/ui
- **Деплой:** Vercel
- **Состояние:** Одностраничный SPA (client-side rendering)
- **Исходники:** `/Users/azarovmaxim/photo2sticker/`

### 1.3. Критическая проблема
Сайт — **SPA с client-side рендерингом**. В HTML отдаётся пустой `<div id="root"></div>`, контент рендерится JS в браузере. Яндекс плохо индексирует JS-контент. Все новые страницы будут невидимы для поисковиков без решения этой проблемы.

**Решение:** Переход на SSG (Static Site Generation) или SSR. Рекомендация — Vite SSG для первой фазы с возможной миграцией на Next.js позже.

### 1.4. Текущая структура сайта

## 6. Техническая реализация

### 6.1. Фаза 0: Миграция на Next.js App Router (критический путь)

**Задача:** Сделать так, чтобы каждый URL отдавал уникальный HTML с title, meta, h1, контентом — без JS-рендеринга.

**Анализ вариантов:**

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Vite SSG** (vite-ssg) | Минимальный рефактор текущего кода | Нет ISR, плохо поддерживается, нет API routes, временное решение |
| **Astro** | SSG по умолчанию, быстрый | Другая парадигма (MPA), нет нативного ISR на Vercel, каждый интерактивный компонент — island |
| **Next.js App Router** | SSR + ISR + SSG + API routes + Server Components + нативный Vercel | Полная миграция (но текущий код маленький) |

**Решение: Next.js App Router.**

Причины:
1. **Vercel-нативный** — нулевой конфиг, лучшая производительность на этом хостинге
2. **`generateStaticParams()`** — генерит 65+ HTML-файлов при билде, идеально для SEO
3. **ISR (Incremental Static Regeneration)** — галереи обновляются без ребилда всего сайта
4. **Server Components** — запрос к Supabase прямо в компоненте, без отдельного API
5. **`metadata` API** — уникальный title/description/canonical/OG на каждой странице нативно
6. **`next/image`** — автоматический WebP, lazy loading, CDN-оптимизация
7. **Текущий стек (React + Tailwind + shadcn/ui)** — переносится 1:1 без изменений
8. **Текущий сайт маленький** (1 страница, ~10 компонентов) — миграция за 2-3 дня

**Структура после миграции:**
```
app/
├── layout.tsx                         ← общий layout (шрифты, Яндекс.Метрика)
├── page.tsx                           ← главная /
├── stiker-iz-foto/page.tsx            ← кластерная
├── bot/page.tsx                       ← кластерная
├── telegram/page.tsx                  ← кластерная
├── iphone/page.tsx                    ← кластерная
├── android/page.tsx                   ← кластерная
├── animirovannye/page.tsx             ← кластерная
├── s-nadpisyu/page.tsx                ← кластерная
├── besplatno/page.tsx                 ← кластерная
├── style/
│   ├── page.tsx                       ← каталог стилей /style/
│   ├── [group]/
│   │   ├── page.tsx                   ← группа /style/anime (generateStaticParams)
│   │   └── [substyle]/
│   │       └── page.tsx               ← подстиль /style/anime/chibi (generateStaticParams)
├── api/
│   └── examples/[styleId]/route.ts    ← API для галерей (фаза 3)
├── sitemap.ts                         ← динамический sitemap
└── robots.ts                          ← robots.txt
```

**Что переносится без изменений:**
- На **кластерных** посадочных выводятся: `Hero`, `PainBlock`, `HopeBlock`, `SocialProof`, `Reviews`, `HowItWorks`, `Features`, блок цены, `FAQ`, Sticky CTA. **Не выводится** `StyleGallery` (его используем на стилевых посадочных). Блок More Pain не используется.
- `StyleGallery` — это **сетка карточек стилей** (превью-стикер + название стиля, grid 3–4 колонки), не карусель и не таблица. На стилевых страницах можно передавать подмножество стилей через props.
- Tailwind CSS + shadcn/ui
- Логика CTA в бота: утилита типа `buildTelegramStartLink(pageSlug?)` (utm_* + yclid из URL при рекламе, иначе `web_{pageSlug}`) и компонент кнопки — см. п. 6.1.1 и 8.2
- Яндекс.Метрика (счётчик 106534984) — см. п. 6.1.1
- Статичные ассеты (images, fonts)

**Что меняется:**
- `App.tsx` → файловый роутинг (`app/page.tsx`, `app/style/[group]/page.tsx`)
- `index.html` → `app/layout.tsx` с metadata API
- `vite.config.ts` → `next.config.js`
- `vercel.json` → упрощается (Next.js сам разруливает роутинг)

### 6.1.1. Перенос с текущего лендинга (обязательно сохранить)

При миграции на Next.js **не потерять** следующее:

| Элемент | Где сейчас | Что перенести |
|--------|------------|----------------|
| **Кнопка перехода в бота** | `TelegramButton.tsx` + `buildTelegramStartLink()` в `landing/client/src/lib/utils.ts` | Логику формирования deep link: при наличии в URL `utm_*` и/или `yclid` — payload из них (для Директа), иначе — `web` или `web_{page_slug}` страницы. На каждой SEO-странице передавать свой `page_slug` в эту функцию. См. п. 8. |
| **Яндекс.Метрика** | `landing/client/index.html` — счётчик `106534984` | Перенести тот же скрипт/тег в `app/layout.tsx` (или в клиентский компонент, подключаемый из layout). Без этого не будет визитов и привязки офлайн-конверсий к кликам. |
| **Парсинг ссылок с Директа** | Лендинг читает `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`, `yclid` из URL и кладёт их в `?start=...` | Сохранить эту логику при переносе `buildTelegramStartLink()`. Конверсии в Метрику шлёт **бот** по факту оплаты (офлайн-конверсии); лендинг только **передаёт** yclid в start — иначе Директ не сможет оптимизироваться на покупки. Подробно: [13-02-yandex-direct-conversions.md](./done/02/13-02-yandex-direct-conversions.md). |

**Итог:** Одна и та же кнопка CTA: для органического захода — `?start=web_{slug}`; для перехода с рекламы (Директ и др.) — payload из UTM + yclid. Реализация: `buildTelegramStartLink(pageSlug)` с приоритетом: если в URL есть utm_source — собирать payload из utm_* и yclid (как в текущем лендинге), иначе возвращать `web` или `web_{pageSlug}`.

### 6.4. Фаза 3: Хранение изображений для сайта

**Проблема:** Стикеры хранятся как `telegram_file_id` — это внутренний ID Telegram, не публичный URL. Для сайта нужны HTTP-ссылки на картинки.

**Решение: Supabase Storage**

1. **Новая колонка в `stickers`:**
```sql
ALTER TABLE stickers ADD COLUMN IF NOT EXISTS public_url text;
```

2. **Изменения в `src/worker.ts`:**
При генерации стикера, после получения PNG:
```typescript
// Upload to Supabase Storage
const { data } = await supabase.storage
  .from('stickers-examples')
  .upload(`${stickerId}.webp`, webpBuffer, {
    contentType: 'image/webp',
    upsert: true,
  });

// Save public URL
const publicUrl = supabase.storage
  .from('stickers-examples')
  .getPublicUrl(`${stickerId}.webp`).data.publicUrl;

await supabase.from('stickers')
  .update({ public_url: publicUrl })
  .eq('id', stickerId);
```

3. **API для сайта:**
```
GET /api/styles/{groupId}/examples?limit=16
→ [{ id, public_url, style_preset_id, created_at }]
```

4. **Когда сохранять:**
- Вариант A: Сохранять ВСЕ стикеры (дорого по хранилищу)
- Вариант B: Сохранять только `is_example = true` ✅ **Принято**
- Вариант C: Сохранять все, но на сайте показывать только `is_example = true`

**Принято:** Вариант B — в Storage попадают только примеры (`is_example = true`).

## 7. Антидубль-стратегия

Каждая страница должна быть **существенно уникальной**, а не перефразированной главной.

### 7.1. Обязательные уникальные элементы

| Элемент | Правило |
|---------|---------|
| `<title>` | 100% уникальный, содержит целевой запрос |
| `<meta description>` | 100% уникальный, ~155 символов |
| `<h1>` | 100% уникальный, отличается от title |
| SEO-текст | 100% уникальный, 400-600 слов |
| FAQ | 100% уникальный набор вопросов из Wordstat |
| Изображения | Разные примеры стикеров на каждой странице |
| `<canonical>` | Указывает на саму себя (не на главную!) |

### 7.2. Переиспользуемые компоненты (допустимо)

- Шапка/футер — одинаковые (нормально)
- Структура "Как это работает" — шаблон одинаковый, **текст шагов разный**
- CTA-кнопка — одинаковый дизайн, **разный текст и UTM**
- TrustBar — одинаковый (нормально для UI-элемента)

### 7.3. Что делает страницу НЕ дублем

Кластерная `/iphone` — это не "главная + слово iPhone". Это:
- **Сравнительная статья:** 3 способа сделать стикер на iPhone (встроенный iOS Live Photo, приложения, наш бот)
- **Скриншоты:** реальные скрины на iPhone
- **FAQ из Wordstat:** "Где хранятся стикеры из фото на айфоне?", "Как удалить стикер?"
- **Специфичные инструкции:** "Откройте Telegram на iPhone → найдите бота..."

Стилевая `/style/anime` — это не "главная в стиле аниме". Это:
- **Каталог подстилей** с превью и описаниями
- **Галерея реальных примеров** этого стиля
- **Статья про аниме-стиль:** история, особенности, какой подстиль выбрать
- **FAQ:** "Сохранятся ли черты лица в аниме стиле?"

Чтобы не дублировать блоки между типами лендингов, закреплено, **что где показывается** и **что обязательно уникально**.

## 8. CTA-стратегия и UTM

### 8.1. Deep link формат

Каждая страница ведёт в бота с уникальным start-параметром:

```
https://t.me/Photo_2_StickerBot?start=web_{page_slug}
```

Примеры:
- Главная: `?start=web`
- `/stiker-iz-foto`: `?start=web_foto`
- `/bot`: `?start=web_bot`
- `/style/anime`: `?start=web_anime`
- `/style/anime/chibi`: `?start=web_anime_chibi`
- `/iphone`: `?start=web_iphone`

### 8.2. UTM и yclid (Директ)

Если пользователь пришёл из рекламы — в URL есть `utm_source`, `utm_medium`, `utm_campaign`, `utm_content` и при клике из Директа — **`yclid`**. Всё это нужно **прокидывать в start-параметр** (не терять при переходе в бота), иначе офлайн-конверсии не привяжутся к клику и Директ не сможет оптимизироваться на покупки.

**Правило:** при наличии в URL хотя бы `utm_source` (или `yclid`) формировать payload из utm_* + yclid (формат и тримминг 64 символа — как в текущем `landing/client/src/lib/utils.ts`). Иначе — `web` или `web_{page_slug}` для данной страницы.

См. [13-02-yandex-direct-conversions.md](./done/02/13-02-yandex-direct-conversions.md): лендинг только передаёт данные в deep link; конверсии в Метрику отправляет бот по факту оплаты.

### 8.3. Текст CTA по контексту

| Страница | Текст кнопки |
|----------|-------------|

| `/stiker-iz-foto` | Превратить фото в стикер |

## 12. Открытые вопросы и принятые решения

1. ~~**Сколько примеров is_example = true в базе?**~~ — **РЕШЕНО:** Страницы и sitemap выводятся для **всех** стилей/подстилей (по данным `style_presets_v2`). Флаг `is_example` влияет только на то, показываем ли стикер **в блоке галереи/примеров** на странице: в блоке — только `is_example = true`; если таких нет — фоллбэк на статичные картинки.
2. ~~**Next.js vs Vite SSG**~~ — **РЕШЕНО: Next.js App Router** (см. раздел 6.1)
3. ~~**Бюджет на Supabase Storage**~~ — **РЕШЕНО:** Пока сохраняем в Storage только примеры (`is_example = true`), см. п. 6.4.
4. ~~**Приоритет контента**~~ — **РЕШЕНО:** Начинаем с кластерных посадочных (выше объём запросов, проще контент).
5. ~~**OG-изображение**~~ — **РЕШЕНО:** Одна общая картинка для всех страниц (указано в чеклисте п. 6.7).
